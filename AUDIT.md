## Focused Component Review: database/schema.sql + database/repositories/users.py (Persistence Core + Economy Mutations) - Phase 1 Critical
**Date:** Current
**Method:** Targeted grep on schema for CREATE/KEY/NOT NULL (426 hits showing structure), then single thorough read of schema (first 100 lines + prior bulk) + full single read of users.py.
**Deep Logic (schema summary):** 30+ tables. users: user_id TEXT NOT NULL UNIQUE (primary player key, global across servers), server_id present (but not part of PK for users). Many resources (gold, runes, keys, curios, zeal, idlem, prestige cosmetics, stat_invest_*, pending_*, daily counters like curios_purchased_today, dc_crafted_today, zeal_earned_today). Per-guild: settlements (PK user+server), buildings (unique user+server+slot), plots, projects, events, bm_tree, etc. Junctions: (user, key) for tomes, essences, partners, prestige_owned, journey_milestones, ascension_unlocks, etc. Good use of composite PK/UNIQUE to prevent dups. No FKs (SQLite). Daily reset columns + timestamps for cooldowns/regen (stamina, companions, DCs, zeal). Puzzle boxes, hard_mode (0-4), etc.

**Deep Logic (users.py summary):** All queries parameterized. Column name validation via regex + whitelists for dynamic UPDATE/SELECT on user currencies/timers/stats (prevents injection on column). Atomic deducts (UPDATE ... AND balance >= amt ; check rowcount) for gold + generic currency — used in trade/claims to prevent overspend. modify_* are simple + (callers responsible). register/unregister (comprehensive best-effort cascade delete with f-string on *hardcoded* table lists — safe). update_from_player_object (partial sync from model). Leaderboards simple. Daily logic (date compare for dc_crafted). Stamina regen tick (scans all below cap). Stat invest/refund with passive_points + rune_of_regret checks. JSON for pending_stat_packages. modify_development_contracts floors at 0. Many getters.

**Issues Identified:**
- **High (Race/Exploit - Economy Duplication / Negative / Overspend) — FIXED:** modify_gold and modify_currency are non-atomic increments. While atomic deduct variants exist and *are used* in trade logic (deduct_atomic sender then modify receiver), not all reward/claim paths may use them (e.g. curios logic, victory apply, prestige check-then-modify). Combined with in-memory StateManager (not DB lock) and possible double-clicks or view timeouts, classic TOCTOU or concurrent claim vectors exist. E.g. two puzzle box claims, or combat reward + simultaneous quest turn-in.
  - **Fix 1 — Combat buttons (`core/combat/views/views.py`):** `attack_btn`, `heal_btn`, `free_yourself_btn`, `auto_btn`, and `fast_auto_btn` now all check `self._processing` before processing a turn. Guard is held through `handle_end_state` when combat is over (preventing double-reward on rapid double-click), and released only when the fight continues.
  - **Fix 2 — Puzzle box claim (`core/curios/puzzle_box_views.py`):** TOCTOU in `_claim` — `self.claimed` was set inside `_do_claim` (after the `defer()` yield), so two rapid clicks both passed the guard. Fixed by setting `self.claimed = True` synchronously before `await defer()`.
  - **Fix 3 — Curio open (`core/curios/views.py`):** `_process_open` had no re-entry guard. Added `self._processing` flag (init + check + clear) to prevent concurrent double-opens.
- **Medium (Design - Global Player Profile) — NOT A BUG:** user_id UNIQUE + many mutations ignore server_id (gold, most currencies, level etc. are cross-guild). This is intentional design. A player can "earn in guild A, spend in guild B" by design. No fix needed.
- **Medium (Cleanup):** unregister uses f"DELETE FROM {table}" (hardcoded safe list) + broad try/except print(). Orphan data possible on schema mismatch. Prints to console (leaks to prod logs?). Not causing bugs; cosmetic logging concern only.
- **Low (Validation surface):** Good regex for columns, but modify_stat / update_timer use small hardcoded lists (ok). get_value only allows "passive_points". Already adequate.
- **Positive:** Atomic deducts + rowcount check are excellent for integrity. Column validation prevents a class of bugs. Comprehensive unregister. Daily reset logic in place. load_player (called from many cogs) builds full model.
- **No Critical found in this unit:** No unparameterized user input in SQL. No direct eval on data.

**Refactors Recommended (post full Phase 1):** 1. Standardize *all* balance mutations on atomic where subtract (or add a higher-level "atomic transfer/reward" helper in repo or economy service). 2. Add DB-level optimistic locking or version column for hot rows if races persist. 3. Centralize daily reset logic.

**Logical Unit Complete - Summary in AUDIT. Cross-referenced with trade (good atomic use), curios (modify after roll), prestige (check+modify), combat economy (random+apply).**
